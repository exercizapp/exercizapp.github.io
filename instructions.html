Updating Exercises on Cloudflare

The standard (library) exercises the Exerciz app renders. Follow this every time.
Where the data lives. Standard exercises are static files in the Cloudflare R2 bucket exerciz-library (served via CDN). They are not in Firebase — Firebase only holds user-created custom exercises. So every change here means changing files in R2.

0 · The golden rule

Every exercise is described by one JSON doc, but it also appears in 4 index files that must always agree:
  1. exercises/{section}/index.json — the section list
  2. exercises/index.json — the top-level manifest (holds libraryVersion)
  3. master-index.json — the flat all-section list (holds media[] + masterIndexVersion)
  4. (the per-exercise doc itself)
If these drift, the app shows stale or missing exercises. The admin UI keeps all of them in sync automatically on every save. If you ever edit by hand, you must update all four — or run the UI's Repair indexes button afterward.

Sections & ID bands (an exercise's section = its folder; the ID just has to fall in the band):

SectionID bandFolder
yoga0000–1999exercises/yoga/
bodyweight2000–3999exercises/bodyweight/
freeweights4000–5999exercises/freeweights/
equipment6000–7999exercises/equipment/

1 · Two ways to edit

WayWhen
Admin UI recommendedAlmost always. Point-and-click, validates input, keeps the 4 indexes in sync, has a preview/dry-run and a local sandbox.
Manual CLI (wrangler)Bulk imports, scripted regeneration, or when the UI isn't deployed.

2 · The admin UI (recommended)

The admin app lives in the repo at admin/. It's a Cloudflare Worker that serves the UI and writes to R2, gated by Cloudflare Access to the authorized admin account.

Open it

  1. Go to the deployed admin URL (the workers.dev address from npm run deploy, or your custom domain) and sign in with the authorized admin account.
  2. Decide your safety level using the badge + toggle at the top-left:
    • Live · production means writes hit the real bucket.
    • Preview changes before saving (on by default) → every action first shows a dry-run diff (which files change, version bumps) and writes nothing until you click Apply changes.
Practice first? Run it locally against a sandbox that can't touch production — see the sandbox box below.

2a · Edit an exercise

  1. Pick the section tab, then click the exercise in the list (search by name/code/body part).
  2. Change fields in the form. Roles are checkboxes; taxonomy and the cross-reference arrays (similar/substitutions/progressions/regressions) are raw-JSON boxes.
  3. Click Save. With preview on, review the diff modal, then Apply changes.
  4. The toast shows the new library v… — that's your confirmation the indexes were bumped.

2b · Add an exercise

  1. Click + New exercise (it creates against the currently selected section).
  2. Fill in at least code (a lowercase slug, no spaces/_/-) and name (Title Case). The ID is assigned automatically as the next free number in that section's band.
  3. Set roles, difficulty, category, instructions, and the default routine (sets, durations, weights).
  4. Click Create → review preview → Apply. Then upload media (next section).

2c · Delete an exercise

  1. Open the exercise, click Delete.
  2. The preview modal lists the doc and every media file that will be removed, plus index updates.
  3. Apply changes to commit. (With preview off, you get a plain confirm dialog instead.)

2d · Upload / replace media

Each exercise has a media card per instructor model (m1 / Marcus, f1 / Nadia) for three assets:

AssetFormatSpec
Video.mp43:4 portrait, HEVC/H.265, 30fps, no audio, ~1–3 MB, short loop
Still.png1080×1440 (3:4) fallback image
Thumb.jpg720×960 (3:4) poster / list thumbnail
  1. Open the exercise → Media section → click Upload (or Replace) on the right card.
  2. Uploading a video/still automatically flips that exercise's hasVideo/hasImage flag.
  3. Media should exist for all models (m1 and f1). The UI warns on an incomplete set but lets you proceed — finish the missing model when you can.

Want to practice safely? Use the sandbox

From the admin/ folder:

export NVM_DIR="$HOME/.nvm"; . "$NVM_DIR/nvm.sh"
npm install
npx wrangler login        # one-time

npm run seed:local        # load real library data into a LOCAL fake bucket
npm run dev               # http://localhost:8787 — badge reads "Sandbox (local)"

npm run dev never touches production. npm run dev:remote uses an isolated cloud bucket exerciz-library-sandbox. Only npm run deploy + the live URL write to production.

3 · The manual CLI way (wrangler)

Origin staging lives at repo-root exercise-library/ and maps 1:1 to the bucket. After editing files there, upload with wrangler. You must keep the 4 indexes in sync yourself (id+code first keys, Title Case name, role array everywhere).

  1. Add/edit the doc at exercises/{section}/{id}-{code}.json and update that section's index.json (bump its version + count).
  2. Update the section entry (version/count) in top-level exercises/index.json and bump libraryVersion.
  3. Update the row in master-index.json and bump totalExercises + masterIndexVersion.
  4. Upload each changed file (note --remote is REQUIRED, else it hits the local sim): 
    export NVM_DIR="$HOME/.nvm"; . "$NVM_DIR/nvm.sh"
npx wrangler r2 object put "exerciz-library/exercises/freeweights/4000-benchpressflat.json" \
  --file "exercise-library/exercises/freeweights/4000-benchpressflat.json" \
  --content-type application/json --remote
  5. Media — upload the video, the -thumb.jpg poster, and the .png still (both images live under media/images/), then add the real filenames to that row's media[] in master-index.json: 
    npx wrangler r2 object put "exerciz-library/media/videos/m1-4000-benchpressflat.mp4" \
  --file ".mp4" --content-type video/mp4 --remote
  6. To remove a renamed/renumbered object: 
    npx wrangler r2 object delete "exerciz-library/exercises/freeweights/<oldkey>.json" --remote
Tip: the 4 indexes are mechanically derivable from the docs + the media/ listing. Prefer the admin UI's Repair indexes (or a regeneration script) over hand-editing once the library grows.

4 · Verify it went live

  1. Open the top manifest in a browser and confirm libraryVersion changed:
    {CDN_BASE}/exercises/index.json (your library CDN base URL)
  2. Check the section index count / your row in …/exercises/{section}/index.json and …/master-index.json.
  3. For media, hit the exact CDN URL, e.g. …/media/videos/m1-4000-benchpressflat.mp4 (should not 404).
  4. In the app: the client refetches a section when its version is newer than its cache, so a version bump is what makes changes appear. (CDN edge cache may lag a little.)
Custom domain note. The base URL is the r2.dev address today. When cdn.exerciz.app goes live, only that base string changes — paths stay identical.

5 · Troubleshooting

SymptomFix
App shows old data after an editConfirm libraryVersion (and the section version) actually bumped. CDN edge cache can lag a few minutes.
Exercise missing / counts look wrongIndexes drifted — run Repair indexes in the UI (regenerates all 4 from the docs + media).
Media 404s but app claims it has videohasVideo/hasImage are claims; the file may not be uploaded. Upload it, or it falls back to image → text.
"validation" error on saveCheck: id in band, code is a lowercase slug, name present, role is a non-empty array, taxonomy.equipmentCategory + defaultRoutine present.
Can't sign inAccess is restricted to the authorized admin account. Use that identity (One-time PIN or Google).
CLI upload "succeeded" but nothing changed onlineYou forgot --remote — without it, wrangler writes to the local simulator, not the cloud.
